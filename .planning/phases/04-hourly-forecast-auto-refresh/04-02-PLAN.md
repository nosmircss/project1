Create `src/renderer/src/hooks/useInterval.ts` implementing the Dan Abramov `useInterval` pattern from RESEARCH.md.

```typescript
import { useEffect, useRef } from 'react'

/**
 * Declarative setInterval hook. Callback ref stays fresh (no stale closures).
 * Pass delay=null to pause the interval without clearing/re-creating it.
 * Source: overreacted.io/making-setinterval-declarative-with-react-hooks/
 */
export function useInterval(callback: () => void, delay: number | null): void {
  const savedCallback = useRef<() => void>(callback)

  // Always keep ref current — runs after every render
  useEffect(() => {
    savedCallback.current = callback
  })

  useEffect(() => {
    if (delay === null) return
    const id = setInterval(() => savedCallback.current(), delay)
    return () => clearInterval(id)
  }, [delay])
}
```

This is a small focused file. The key behaviors:
1. `savedCallback` ref ensures the interval always calls the latest callback (no stale closure)
2. The ref update effect has NO dependency array — it runs after every render to keep the ref fresh
3. The interval effect only re-runs when `delay` changes — NOT when callback changes (that's the whole point)
4. Passing `delay = null` causes early return (no interval created) — this is how pause works

**New UseWeatherResult interface:**
```typescript
interface UseWeatherResult {
  weather: WeatherData | null
  hourly: HourlySlice[]           // 12 entries from current hour (empty if no data)
  loading: boolean                // true only when weather === null (first fetch for location)
  isRefreshing: boolean           // true during background auto-refresh (no skeleton trigger)
  error: string | null
  lastUpdatedAt: Date | null      // timestamp of last successful fetch for active location
  nextRefreshAt: number | null    // Date.now() ms target for next refresh
  refetch: () => void
}
```

**Core implementation requirements:**

1. **Per-location cache** — Store a `Map<string, { data: WeatherData; lastUpdatedAt: Date }>` as a `useRef`. When switching locations:
   - If cached data exists for the new zip, immediately display it with its original `lastUpdatedAt` timestamp
   - Then trigger a background refresh (isRefreshing=true, NOT loading=true) if the data is older than the refresh interval
   - If no cache exists, set `loading=true` and show skeleton (existing behavior)
   Per locked decision: "switching back to a previously viewed location shows how stale that data is and triggers a refresh"

2. **Hourly data slicing (HOUR-02)** — After a successful fetch, slice the raw `weather.hourly` parallel arrays into `HourlySlice[]`:
   ```typescript
   function sliceHourly(hourly: HourlyData | undefined): HourlySlice[] {
     if (!hourly) return []
     const now = new Date()
     const currentHourISO = `${now.getFullYear()}-${String(now.getMonth() + 1).padStart(2, '0')}-${String(now.getDate()).padStart(2, '0')}T${String(now.getHours()).padStart(2, '0')}:00`
     const startIdx = hourly.time.findIndex(t => t >= currentHourISO)
     if (startIdx === -1) return []
     return hourly.time.slice(startIdx, startIdx + 12).map((time, i) => ({
       time,
       temperature: hourly.temperature[startIdx + i],
       weatherCode: hourly.weatherCode[startIdx + i],
       precipProbability: hourly.precipProbability[startIdx + i]
     }))
   }
   ```
   Per RESEARCH.md Pitfall 4: the hourly time array starts at midnight local time. `findIndex(t => t >= currentHourISO)` finds the current hour.

3. **Auto-refresh via useInterval (REFR-01)** — Use the `useInterval` hook:
   ```typescript
   const intervalMs = settings.refreshInterval * 60_000
   const delay = (location && isWindowVisible) ? intervalMs : null
   useInterval(autoRefreshCallback, delay)
   ```
   - `delay = null` when no location or window not visible (pauses interval)
   - The `autoRefreshCallback` calls the fetch function with `isRefreshing=true` (NOT `loading=true`)
   - Per locked decision: "Data updates silently in place — no loading skeleton or flash on auto-refresh. Numbers just change."

4. **Silent refresh (no skeleton flash)** — Introduce `isRefreshing` state:
   - On initial fetch (weather === null): `setLoading(true)` — shows skeleton
   - On auto-refresh (weather !== null): `setIsRefreshing(true)` — no skeleton, data stays visible
   - On success: update `weather` state directly (numbers just change), update cache, update `lastUpdatedAt`
   - Per RESEARCH.md Pitfall 6: NEVER set loading=true on auto-refresh

5. **Failed refresh retry (locked decision)** — If auto-refresh fails while existing data is visible:
   - Keep stale data
   - Set error message ("Could not refresh weather. Data may be outdated.")
   - Set `nextRefreshAt` to `Date.now() + 30_000` (retry in 30 seconds instead of full interval)
   - The useInterval will fire at the normal interval, but the callback checks `nextRefreshAt` and skips if not yet due. SIMPLER alternative: temporarily change the interval delay to 30000, then restore after successful retry.
   - Actually, simplest approach: after failed refresh, set a `retryDelay` ref to 30000. The useInterval delay becomes `retryDelay.current ?? intervalMs`. On success, reset retryDelay to null.

6. **Visibility pause/resume (locked decision)** — Subscribe to window visibility via `window.electronAPI.onWindowVisibility`:
   ```typescript
   const [isWindowVisible, setIsWindowVisible] = useState(true)
   useEffect(() => {
     const cleanup = window.electronAPI.onWindowVisibility((visible) => {
       setIsWindowVisible(visible)
       if (visible && location) {
         // Check if interval elapsed while hidden — if so, refresh immediately
         if (nextRefreshAtRef.current && Date.now() >= nextRefreshAtRef.current) {
           autoRefreshCallback()
         }
       }
     })
     return cleanup
   }, []) // stable — uses refs inside
   ```
   Per locked decision: "Resumes immediately when window becomes visible, triggering an immediate refresh if interval has elapsed"

7. **nextRefreshAt tracking (REFR-03)** — After each successful fetch, set:
   ```typescript
   const nextTarget = Date.now() + intervalMs
   setNextRefreshAt(nextTarget)
   nextRefreshAtRef.current = nextTarget
   ```
   Use both state (for downstream display) and ref (for visibility resume check). Per RESEARCH.md Pitfall 2: set `nextRefreshAt` after fetch completes, not when interval fires.

8. **Location switch behavior (HOUR-03, locked decisions)** — When `location` changes (detected via zip):
   - Check per-location cache for existing data
   - If cached: set weather from cache, set lastUpdatedAt from cache, set isRefreshing=true (background refresh)
   - If not cached: clear weather to null, set loading=true (skeleton)
   - Reset nextRefreshAt
   - Trigger immediate fetch
   - Per locked decision: "Auto-refresh timer resets on location switch — immediate fetch + fresh countdown"

9. **Keep existing fetchWithRetry** — The retry-with-backoff function stays for the initial fetch. Auto-refresh should NOT use retry — it fires once, and if it fails, waits 30s for the next attempt.

**CRITICAL: Preserve backward compatibility.** The return type adds new fields but keeps `weather`, `loading`, `error`, `refetch` in the same shape. Plan 04 will update consumers to use the new fields. Until then, existing code still works because `loading` only flips true on first fetch (same behavior as before).
